Code Review Checklist
This skill provides a structured, language-agnostic code review workflow for any codebase. It guides the agent through multiple passes to focus on correctness, security, code quality, performance, testing, and maintainability aspects of the code.
The review prioritizes critical issues (e.g. major bugs or security vulnerabilities) first, then important improvements, and finally minor suggestions. It also encourages highlighting good practices with positive feedback[1] to reinforce what the code is doing well. The output is a clear, well-organized report with each finding labeled by severity and referencing specific files/lines for context.
Workflow
1.	High-Level Pass: Focus on any showstopper issues that would prevent safe release. Scan for critical bugs, logic errors, or security vulnerabilities. Examine the overall design for major flaws or misalignments with requirements. (This addresses the most severe problems first.)
2.	Detailed Pass: If the high-level check is clear, perform a thorough line-by-line review. Check maintainability, code style, clarity, performance, and compliance with best practices. Identify important issues that should be fixed for long-term quality.
3.	Final Pass: Do a final sweep for minor issues or nitpicks (small styling issues, minor optimizations). Also, note any positive aspects or improvements in the code. By deferring these to the end, the review stays constructive and focuses on essentials first.
This multi-pass approach ensures critical problems are addressed before spending time on minor details, aligning with the principle that code reviews should seek continuous improvement rather than perfection[2].
1. Architecture & Design
•	Problem & Requirements: Does the code solve the intended problem and meet all requirements? Is the approach logically correct for the stated goals?
•	Approach Appropriateness: Is the solution approach appropriate for the scope and constraints of the problem? (Not over-engineered for a simple task, and not overly simplistic for a complex requirement.)
•	Consistency with Architecture: Does the change fit with the existing system design and architecture patterns? It should not introduce a radically different design without good reason.
•	Trade-offs & Design Decisions: Are major design decisions or trade-offs well-reasoned and documented? For example, using a certain algorithm or pattern should have justification (performance vs. readability, short-term vs. long-term benefits, etc).
•	Technical Debt: Does this change introduce technical debt or cleanup existing debt? If it introduces debt (e.g. a quick workaround), is it acknowledged via comments or TODOs for future refactoring?
•	Rollout & Future Impact: For large or breaking changes, is there a rollout or migration plan? Consider backward compatibility, feature flags, or incremental rollout strategies to mitigate risk. Also, think about the long-term implications: will this change be easy to modify or extend later?
Red flags: Solving a symptom rather than the root cause of a problem, an unnecessarily complex solution when a simpler one would suffice, diverging from established architecture without rationale, or lack of consideration for how the change will be maintained in the future.
2. Correctness & Logic
•	Functionality: Does the code do what it’s supposed to do? Verify that the logic aligns with the requirements and that there are no functional bugs.
•	Edge Cases: Are edge cases and error conditions handled? This includes empty or null inputs, boundary values (e.g. off-by-one issues), exceptions, and any unusual scenarios that could occur. Failing to handle edge cases can lead to crashes or incorrect behavior.
•	Error Handling: Does the code handle errors gracefully and informatively? (No silent catch of exceptions without logging or user notification, no ignoring of return codes.) Errors and exceptions should be caught where appropriate and either handled or propagated with context.
•	Concurrency & Async: If the code is asynchronous or multi-threaded, are potential race conditions or timing issues addressed? Ensure that resources are properly synchronized or awaited, and that timeouts or cancellations are considered where relevant.
•	Data Integrity: Check that algorithms and calculations are correct (for example, no integer overflow issues, correct use of order of operations, proper indexing in loops). If the code modifies data or state, ensure it doesn’t corrupt data or leave things in an inconsistent state.
Red flags: Missing null/undefined checks, assuming the "happy path" will always execute (and not handling when it doesn’t), logic that contradicts the specifications, off-by-one errors in loops or array indexing, and any scenario where a failure isn't handled (which could lead to crashes or data corruption).
3. Security
•	Input Validation: Are all inputs (especially from users or external sources) validated or sanitized? Unvalidated input can lead to vulnerabilities like injections. For instance, directly embedding user input in a SQL query or command is dangerous[3] – the code should use safe patterns (e.g. parameterized queries or prepared statements).
•	Authentication & Authorization: Does the code properly enforce security checks? Verify that any operation requiring user credentials or permissions actually performs those checks. For example, a web handler that deletes user data should confirm the requester is authorized.
•	Secrets Management: Ensure that no secrets or credentials are exposed in the code. API keys, passwords, tokens, or any sensitive info should be stored securely (environment variables, vault, etc.), not hard-coded in plain text.
•	Use of Secure Practices: The code should follow known security best practices for its context. This might include using HTTPS for network calls, proper cryptographic practices (no broken or deprecated algorithms), protecting data at rest and in transit, and avoiding common pitfalls from the OWASP Top 10[3][4] (such as XSS by escaping outputs, preventing SQL injection, avoiding open redirects, etc.).
•	Dependency Risks: Check if the code relies on libraries or dependencies with known vulnerabilities. Using an outdated library version that has security issues can introduce risk. Ensure updates or mitigations are in place if so.
Red flags: Direct use of untrusted data in queries/commands (e.g. building SQL strings directly from user input), missing authorization checks on sensitive functions (e.g. no permission check before performing admin actions), any occurrence of TODO: security left unaddressed, hard-coded credentials or cryptographic keys in the code, and failure to sanitize outputs that could be rendered in a browser (risking XSS).
4. Code Quality & Readability
•	Naming & Clarity: Are variables, functions, and classes named clearly and meaningfully? Names should reveal intent (e.g. calculateChecksum vs. func1). Unclear or misleading names hinder understanding. Following standard naming conventions is important.
•	Single Responsibility: Does each function or module do one thing and do it well? Functions should not be too large or handle completely unrelated tasks. This aligns with clean code principles (each function “should do one thing... only.”[5]). If a function seems to be doing too much, consider if it should be refactored.
•	Simplicity: Is the code as simple as possible for the given task? Simpler code is easier to read and less prone to bugs. Watch for overly complex logic, deeply nested loops/conditionals, or redundant code. Sometimes a more straightforward solution exists than the one implemented.
•	Consistency & Style: Is the code formatted consistently and according to the project's style guidelines? Check indentation, bracket placement, spacing, etc. Inconsistent style can be distracting. Also ensure usage of language idioms and best practices (e.g. avoid unnecessary global variables, prefer clear control structures).
•	Avoid Magic Numbers: Are there magic numbers or strings in the code? Any literal value that has significance should be replaced with a named constant or configuration. This makes code self-documenting. For example, seeing if (speed > 136) is less clear than defining MAX_SPEED = 136 and using if (speed > MAX_SPEED).
Red flags: Extremely long functions or files (could indicate they are doing too much), variables with meaningless names (i, data, obj in a broad scope) or misnamed (name doesn't match what it represents), duplicated code blocks that suggest an abstraction is needed, and sections of code that are hard to follow due to complexity or poor organization.
5. Performance & Efficiency
•	Algorithmic Efficiency: Check for obvious inefficiencies. Are there any O(n²) or worse algorithms where n could be large? For instance, be wary of N+1 query patterns in database access (e.g. querying inside a loop for each item[6]) – such patterns can often be refactored to batch queries or use joins. Ensure loops or recursion will terminate and are not unnecessarily repeated.
•	Resource Management: Does the code manage memory and resources efficiently? For example, large data structures that grow without bounds, or not closing file/network connections can lead to leaks. Ensure that any heavy computation or resource usage is justified and, if possible, optimized (e.g. using streaming for large files instead of loading entirely into memory).
•	Scalability Considerations: If the code will run in a context with large workloads (large input sizes, high concurrency, etc.), has it been written with scalability in mind? This includes using pagination or lazy loading for large data sets, caching frequently used results, and avoiding blocking operations on critical paths.
•	Client-Side Performance: For front-end code, ensure that expensive computations are not done on the main UI thread in a way that would freeze the interface. Utilize throttling/debouncing for high-frequency events and optimize DOM updates (for example, batch DOM changes or use requestAnimationFrame for visual updates).
•	Avoid Premature Optimization: Note if the code is overly optimized in a way that sacrifices clarity without a proven need. While performance is important, it should be balanced with maintainability. If an optimization is complex, there should be a comment or documentation explaining its necessity (often with a measurement or rationale).
Red flags: Database queries inside loops without limiting results, lack of indexing on database queries that filter large tables, doing costly operations (like regex or parsing) repeatedly instead of caching results, not releasing resources (file handles, connections), and any obvious bottleneck that would degrade performance as data volume grows.
6. Testing & Coverage
•	Test Coverage: Are there tests covering the new code or changes? Important logic should have corresponding unit tests or integration tests. If this is a bug fix, there should ideally be a test that fails before the fix and passes after, to prevent regression.
•	Test Scope: Do tests cover edge cases and error conditions as well as the normal cases? Good tests will intentionally feed bad or extreme inputs to ensure the code handles them (e.g. null values, extremely large numbers, unexpected sequences of calls).
•	Test Quality: Are the tests written in a clear and maintainable way? They should be checking behavior (outputs, state changes, UI effects) rather than implementation details. For example, a test for a function should verify the result of the function, not whether a certain internal function was called (unless absolutely necessary). This ensures tests don’t break with minor refactoring.
•	Automated Testing: If applicable, check that any continuous integration (CI) or automated test pipelines pass for this code. A code review should flag if new code is breaking existing tests or if tests are being skipped without justification.
•	Test Naming & Documentation: Test cases should have descriptive names (it’s clear what they are verifying). If a complex scenario is tested, a brief comment in the test can help explain the intent. This makes it easier for future maintainers to understand failures.
Red flags: No tests for new non-trivial code, tests that only cover the simplest path and ignore potential fail cases, tests that frequently produce false positives/negatives (flaky tests), or tests that are so tightly coupled to the code’s internals that they will need to be rewritten for any small code change.
7. Documentation & Comments
•	User-Facing Documentation: If the change adds or modifies a feature, is the relevant documentation updated? This could be a README, user manual, API documentation, or even inline help. Users of the code (or other developers) should be aware of new behaviors or changes. For example, a new API endpoint should come with usage docs or examples.
•	Code Comments: Check that complex or non-obvious code blocks have comments explaining why the code is doing something a certain way. Avoid comments that only explain "what" the code does – the code should make that obvious. But any rationale, especially for unusual workarounds or important decisions, should be captured in comments.
•	Changelog & Migration: For breaking changes or major feature additions, is there an entry in a CHANGELOG or release notes? If the change requires users to do something (like migrate data, update config, etc.), those steps should be documented clearly. A migration guide or deprecation notice is critical if applicable.
•	In-Code Documentation: Are function/method signatures documented (via docstrings or comments) with their purpose, parameters, and return values if it's a public or complex function? This helps others understand how to use the code. Additionally, ensure any new configuration options or environment variables are explained somewhere.
•	Todos and Follow-ups: If there are TODO/FIXME comments left in the code, the review should note whether there's a plan to address them (e.g. a linked issue). While it's fine to leave minor TODOs, anything significant should be tracked so it isn't forgotten.
Red flags: Missing documentation for a new public API or feature, absent comments for complex logic or hacks (making it hard for others to know the intent), outdated comments that no longer match the code, or a breaking change that isn’t mentioned anywhere for others to find.
Severity Levels and Output Format
When writing the review report, categorize findings by severity:
•	Critical – Issues that must be fixed before merge. These typically include bugs that break functionality, cause crashes or data loss, and security vulnerabilities. For example, an authentication bypass bug or a SQL injection risk should be marked as Critical. The reviewer should provide clear explanation why it's critical and how it impacts the system.
•	Important – Significant issues that should be addressed, but might not block merging if there's a compelling reason. These can include logic issues that could lead to bugs in edge cases, lack of test coverage for important scenarios, performance concerns, or deviations from standards/best practices that could maintainability or future development. They are one step below critical, but still merit prompt attention.
•	Suggestion – Minor improvements or optimizations that are nice-to-have. These include style inconsistencies, minor refactoring opportunities, naming improvements, or adding a non-essential test. Suggestions can be fixed at the author's discretion. They often begin with "Nit:" or are phrased as gentle recommendations.
Format the output for clarity: Present the review comments in a way that's easy to read and trace. For example: - List each finding as a separate bullet point (or numbered item), and start with the severity in bold (e.g. Critical:). - For each issue, provide a concise description and rationale. If possible, include the relevant file name and line number(s) in parentheses or brackets for quick reference. For instance: Important: utils.py:120-128 – Potential off-by-one error in loop boundary. - Grouping by severity can be helpful: you might list all Critical issues first, then Important, then Suggestions. This ensures the most severe problems get attention first. - Include a short Positive Feedback section (or intermix positive notes) to highlight what is well-done. Mark these with a 👍 or a label like "Good:" so they stand out. Example: 👍 Good: The use of parameterized queries for database access is excellent, as it prevents SQL injection vulnerabilities. This kind of positive note encourages the author and reinforces best practices[1].
By providing file/line references and a structured list, the author can easily follow up on each point. The goal is to make the review actionable: the author should know exactly what to change or consider for each item.
Additional Guidance
•	Tone and Clarity: Always be respectful and constructive. Phrase comments in a way that explains the issue and possibly how to improve it. Instead of "This is wrong/bad," prefer something like "Consider doing X, because Y (this will improve Z)". The review should feel like a collaboration toward better code, not an attack.
•	Leverage Tools and Skills: If available, take advantage of automated tools to augment the review. For example, if there's a testing skill or CI integration, run the tests to see if anything fails. If a linters or static analysis tool is available, use it to catch style issues or common bugs (but still double-check important security and logic issues manually). These tools can quickly highlight issues like unused variables, suspect patterns, or style deviations which you can include in your feedback.
•	Big Picture and Future Work: Beyond the code at hand, consider the broader context. Does this change have broader implications for the system’s health or future development? If something is out of scope to fix now, you can still mention it as a suggestion or a follow-up task (e.g. "Suggestion: We might want to refactor the XYZ module in the future to improve scalability, though it's beyond this PR's scope"). This surfaces technical debt or risks proactively.
•	Approval Criteria: Remember that the goal of code review is to improve code health over time[2]. Encourage the author to fix critical and important issues; suggestions can be optional. If all critical concerns are resolved and the code overall improves the codebase, it is reasonable to approve the change. Minor imperfections can be tolerated if the overall direction is positive, to avoid nitpicking that delays progress.
