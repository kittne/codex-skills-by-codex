## Configuration Strategy for pyrightconfig.json and pyproject.toml

### Configuration file selection and precedence

Pyright supports configuration in:

- `pyrightconfig.json` (default filename, typically at project root)  
- a `[tool.pyright]` section within `pyproject.toml`

## Type Stubs, Typed Third-Party Libraries, and Import Resolution

### How Pyright resolves imports

Pyright documents a deterministic import-resolution order. For absolute imports, it searches (summarized):

1. `stubPath` (custom stubs)  
2. workspace code (execution environment roots + extraPaths + common `src`)  
3. installed packages (stub packages, inline stubs, `py.typed` inline types, and possibly library code)  
4. typeshed stdlib stubs  
5. typeshed third-party stubs  
6. fallback heuristics ŅłĆciteŅłéturn7view0Ņłü  

This resolution model implies several best practices:

- If your repo uses a `src/` layout, explicitly include it and consider setting `executionEnvironments` root(s) so Pyright doesnŌĆÖt depend on heuristics. ŅłĆciteŅłéturn7view0Ņłéturn4view0Ņłü  
- Prefer ŌĆ£typed installationsŌĆØ (PEP 561 `py.typed` or stubs) over relying on `useLibraryCodeForTypes` inference. Pyright explicitly warns that analyzing `.py` library code is expensive and often yields low-value inference, particularly around generics and dynamic exports. ŅłĆciteŅłéturn7view1Ņłéturn4view0Ņłü  

### Why stubs and `py.typed` matter

PyrightŌĆÖs type stub documentation is unambiguous:

- Stubs (`.pyi`) define the public interface contract.  
- Pyright will try to resolve imports with stubs firstŌĆöand if it cannot locate type stubs for an external import and the package is not `py.typed`, then imported symbols are treated as `Unknown`. ŅłĆciteŅłéturn7view1Ņłü  

This directly impacts strictness: strict mode escalates many ŌĆ£unknown/partial unknownŌĆØ conditions to errors, which can make strict infeasible unless third-party types are in good shape. Maintainer guidance reinforces that strict will report uses of unknown or partially unknown types, and importing from libraries without full type information can block strict adoption. ŅłĆciteŅłéturn14search18Ņłéturn16view0Ņłü  

### Best practices for third-party library typing

PyrightŌĆÖs getting-started guide recommends a pragmatic order:

1. **Update dependent libraries** because many popular libraries have added inlined types in recent versions. ŅłĆciteŅłéturn15view0Ņłü  
2. Enable `reportMissingTypeStubs` and add minimal stubs where required. ŅłĆciteŅłéturn15view0Ņłéturn7view1Ņłü  
3. Look for stub companion packages named with `-stubs` suffix. ŅłĆciteŅłéturn15view0Ņłéturn7view0Ņłü  
4. If none exist, create custom stubs for the subset of API you consume and check them into your repo. ŅłĆciteŅłéturn15view0Ņłéturn7view1Ņłü  

This aligns with PEP 561: stub packages are expected to be named `distribution-name-stubs`, and `py.typed` is the marker for inline typing. ŅłĆciteŅłéturn7view0Ņłéturn9view3Ņłéturn8search12Ņłü

## Actionable Recommendations

### Recommended policies that scale

**Configuration policy**
- Commit a shared config and make it portable (relative paths, no shell-variable paths). ŅłĆciteŅłéturn15view0Ņłéturn4view0Ņłü  
- Explicitly set `pythonVersion` to your minimum supported runtime (3.8ŌĆō3.12 range) or your baseline (e.g., 3.11/3.12) to avoid default drift. ŅłĆciteŅłéturn4view0Ņłéturn21view1Ņłéturn13search15Ņłü

**Strictness policy**

- Treat strict mode readiness as depending on third-party type quality; if a module imports untyped libs, keep it non-strict until you install stubs or write minimal local stubs. ŅłĆciteŅłéturn14search18Ņłéturn7view1Ņłéturn9view3Ņłü  

**Third-party typing policy**

- Use stub generation (`--createstub`) as a bootstrap, not as a finished product; clean up re-exports and decorator signatures. ŅłĆciteŅłéturn7view1Ņłéturn5view1Ņłü  

**CI policy**
- Pin Pyright versions in CI. Use `pyright-action` where possible for speed and annotations. ŅłĆciteŅłéturn7view2Ņłéturn19search3Ņłéturn21view1Ņłü  
- Decide early whether warnings should fail builds (`--warnings`) and treat that as a maturity lever. ŅłĆciteŅłéturn5view1Ņłü
