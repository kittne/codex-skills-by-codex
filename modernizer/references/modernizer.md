Modernizer
Goal
Provide an end-to-end workflow to modernize an existing software project (codebase, architecture, and processes) to current best practices. The skill covers everything from analyzing the current system and identifying outdated patterns to planning a target state and scaffolding a updated project structure, with minimal user input except for validation at key checkpoints.
Workflow
1.	Analyze Current State: Analyze the current codebase, architecture, and tech stack. Identify all components (e.g., frontend, backend, mobile, database), programming languages, frameworks, and how they interact. Note any outdated libraries, known pain points (performance issues, security risks, etc.), and the overall architectural style (monolith, microservices, serverless, etc.).
2.	Document Features & Logic: Document each major feature or module of the application. Create a brief summary of what each feature does and how it’s implemented in the current system (this yields per-feature documentation). Ensure the core business logic and requirements behind each feature are understood and will be preserved during modernization.
3.	Assess Cross-Cutting & Infrastructure: Gather insights on cross-cutting concerns and infrastructure. Review how the system handles concerns like authentication, authorization, logging, error handling, and configuration. Document the infrastructure and operational setup: deployment environment, cloud services or on-prem setup, CI/CD pipelines, build processes, and any third-party integrations. Identify opportunities where modern tools or practices (e.g., improved observability, containerization, infrastructure-as-code) could be introduced.
4.	User Checkpoint – Validate Analysis: Present a summary of the current-state analysis to the user for validation. Summarize the key findings: the tech stack, major features, pain points, and areas that need improvement. Confirm with the user that the understanding of the existing system is correct and that the preliminary modernization goals align with their expectations before proceeding.
5.	Design Target Architecture & Tech Stack: Design the desired future state of the system. Propose a modernized architecture suited to the project’s needs (for example, breaking a monolith into microservices, adopting a modular monolith, or leveraging serverless functions where appropriate). Select modern technologies for each part of the stack – such as choosing a contemporary front-end framework or library, updating backend language/runtime versions, using cloud-native services, etc. Ensure the new architecture and tech stack address current limitations and align with business goals (for instance, consider an orchestrator like Dagster for data pipelines if the project involves complex data workflows).
6.	Plan Code Modernization: Develop a detailed plan for refactoring and upgrading the codebase feature by feature. Break down the modernization into phases or tasks (e.g., tackle one module or layer at a time) and define clear milestones. For each major change, outline the approach – for example, using automated code transformation tools to accelerate refactoring (OpenRewrite for Java-based code, jscodeshift or other codemods for JavaScript/TypeScript, etc.). Ensure that after each phase, the system can be built and tested (maintaining functionality with minimal regressions). Incorporate strategies for backwards compatibility or feature flags to enable incremental integration of modernized components without disrupting the whole system.
7.	Plan DevOps, CI/CD, and Testing Improvements: Plan updates to the development process and infrastructure to support the modernized system. Recommend improvements to CI/CD pipelines and testing strategy: for instance, adopt a monorepo build system (using Nx or TurboRepo) if consolidating projects, introduce containerization (Docker/Kubernetes) for consistent deployments, and ensure automated tests (unit, integration, end-to-end) are updated or added for new changes. Outline how the CI/CD pipeline will build, test, and deploy the refactored components (e.g., set up separate pipelines for microservices or apps). Define a deployment strategy for rolling out the modernized system, including phased rollouts (canary releases, feature toggles, or parallel run of old and new systems) and a rollback plan in case of issues. Additionally, consider improvements to developer experience such as implementing a developer portal or service catalog (e.g., Backstage) to document and manage the new components and workflows.
8.	User Checkpoint – Approve Modernization Plan: Present the complete modernization plan to the user for approval. This includes the proposed target architecture, the breakdown of tasks/phases with timelines and milestones, planned changes to the tech stack and processes, and identified risks or external dependencies. Ensure the user agrees with the plan and make any adjustments based on feedback. The skill should proceed with implementation only after the user has approved the overall modernization approach and scope.
9.	Scaffold New Structure & Documentation: Execute the initial modernization scaffolding and prepare documentation. Generate a new project structure (for example, create a /modernizedone/ directory) reflecting the target architecture, including the foundational directories, configuration files, and placeholder modules/services as outlined in the plan. Provide an overview of this scaffold (e.g., a tree listing of the new project layout). In parallel, produce updated documentation for the project: an updated /docs/README.md that explains the new system architecture and how to set it up, documentation for each feature (detailing how that feature will be implemented or has changed in the modernized system), and additional guides for important front-end, cross-cutting, or infrastructure aspects (for example, how authentication or logging is handled in the new setup, or how to navigate the new repository structure). Present all these outputs to the user for review, forming the basis for the team to begin the modernization implementation.
Output format
Provide the following outputs during and upon completion of the workflow:
•	Analysis Report: A summary of the current system (features, tech stack, issues) for user validation (mid-process).
•	Modernization Plan: A detailed plan document outlining the target architecture, the phases of work (with tasks and milestones), changes to tech stack, CI/CD/testing updates, and identified risks & dependencies.
•	Project Scaffold: A new project directory structure (e.g. under /modernizedone/) initialized with boilerplate code, configuration, and folders for the modernized application. This can be presented as a directory tree or list of key files/folders.
•	Updated Documentation: Include a new /docs/README.md describing the modernized system and instructions, per-feature documentation files (one for each major feature, explaining its new design or implementation), and additional docs for front-end architecture changes, cross-cutting concerns, or infrastructure/deployment details as needed.
Constraints
•	Scope Focus: Only address tasks related to modernizing the existing project. Do not implement unrelated new features or answer questions outside of modernization/refactoring context.
•	No Context Assumption: Do not assume information that is not provided. If important details about the project are missing (architecture, requirements, etc.), ask the user or infer from the codebase context rather than guessing.
•	Preserve Intent and Functionality: Ensure that the modernization plan retains all required business functionality. Modernize the implementation and architecture without losing features or altering expected behaviors (unless specified by the user).
•	Iterative Engagement: Follow the step-by-step workflow and engage the user at the designated checkpoints. Do not skip directly to final output without presenting the analysis and plan for confirmation. Pause for user approval after summarizing the analysis (Step 4) and after proposing the plan (Step 8) before moving forward.
•	Adapt to Context: Tailor the modernization suggestions and tools to the project’s tech stack and domain. If certain steps or tools are not applicable (e.g., no mobile component or monorepo is not needed), adjust accordingly and focus on relevant areas.
•	Safe Changes: Any code or architecture changes proposed should minimize risk: use feature flags, backward compatibility, and thorough testing strategies to ensure the existing system can transition safely to the new one. Plan for rollback in case a modernization step fails.
Example triggers
•	"Modernize our legacy web application to current standards."
•	"Our system is outdated – help us refactor and update the entire stack."
•	"We want to migrate this monolithic app to a modern microservices architecture."
•	$modernizer
