Configuring NGINX as a Secure Mail-Aware Reverse Proxy
NGINX’s mail proxy module allows it to serve as a single secure entry point for email protocols (SMTP, IMAP, and POP3) while routing traffic to internal mail servers like Postfix, Dovecot, or Exim[1]. This setup can simplify your mail deployment by centralizing TLS termination, load balancing multiple back‑end servers, and applying security controls at the network edge. The following guide details best practices for configuring NGINX as a mail-aware reverse proxy, including support for common mail protocols, authentication methods, and security hardening measures. Clear examples and configuration snippets are provided to illustrate each aspect.
Overview of NGINX Mail Proxy and Benefits
Single Endpoint & Load Balancing: When configured as a mail proxy, NGINX listens on standard mail ports (e.g. 25, 143, 587, 993, etc.) and proxies SMTP, IMAP, and POP3 connections to one or more upstream mail servers[1]. This creates a single mail gateway (e.g. mail.example.com) that clients connect to, which can then distribute connections to internal servers based on defined rules. Benefits include the ability to scale out mail servers easily and load balance traffic across multiple hosts. For example, you can route users to the nearest or least-loaded mail server transparently[1]. It also enables seamless migrations or failovers – clients always hit NGINX, which decides the correct back-end (as illustrated later).
Edge Security and SSL Offloading: Placing NGINX at the WAN edge lets you enforce security policies and handle TLS encryption in one place. NGINX can terminate SSL/TLS on behalf of mail servers, presenting a trusted certificate and then proxying unencrypted traffic internally (or re-encrypting to the LAN if desired). This offloads heavy encryption work from your mail servers and ensures consistent TLS settings (protocol versions, ciphers) for all services. It also means only NGINX’s IP/port needs to be exposed externally – your actual mail hosts can reside on a protected LAN with restricted access.
Protocol Awareness: Unlike a generic TCP proxy, NGINX’s mail module understands mail protocols enough to manage authentication and STARTTLS handshakes. It can advertise protocol capabilities and handle initial auth negotiation with the client (via an external auth service), then hand off the session to the appropriate mail server. This awareness allows for features like selecting an upstream based on the user’s email or domain, and preventing unauthorized use (e.g. stopping open-relay attempts) at the proxy level.
Preparing NGINX for Mail Proxying
Before configuring the mail proxy, ensure your NGINX installation supports the mail modules. NGINX Open Source requires compilation with the --with-mail module (and --with-mail_ssl_module for TLS support)[2]. Many distro packages include this (for example, the nginx-full package on Debian/Ubuntu comes with mail support). You can verify by running nginx -V and checking for --with-mail_ssl_module in the output[3][4]. NGINX Plus has mail proxy functionality built-in by default[5].
Create the Mail Context: Mail proxy settings live in a top-level mail { … } block in NGINX’s config (at the same level as the http { } block)[6]. Inside this context, you define global mail proxy parameters and one or more server blocks for each listener (protocol/port combination). At minimum, set a server_name for the proxy (this is used for protocol banners and TLS certificate selection)[7]. For example:
mail {
    server_name  mail.example.com;
    # ... other settings ...
    server {
        listen     993;
        protocol   imap;
        ssl        on;
        # ... 
    }
    server {
        listen     25;
        protocol   smtp;
        starttls   on;
        smtp_auth  none;
        # ...
    }
    # ... (additional server blocks for POP3, SMTP submission, etc.)
}
Each server block specifies a listening port and the mail protocol for that port[8]. NGINX can infer protocol from standard port numbers, but it’s good practice to be explicit using the protocol smtp|imap|pop3; directive[8]. You should also declare what authentication methods are allowed on that listener via smtp_auth, imap_auth, or pop3_auth directives[9]. For example, for an SMTP submission port you might allow login and plain (two common SASL methods) whereas on an MX inbound SMTP port you might set smtp_auth none to disable client AUTH on port 25[10]. NGINX will advertise and accept only the specified auth mechanisms on that connection. Below is a sample snippet defining authentication on various protocols:
•	SMTP (Port 25) – listening on port 25 with protocol smtp and smtp_auth none (no AUTH for inbound mail)[10]. This prevents misuse of the MX port for client submission.
•	SMTP Submission (Port 587) – could be configured on port 587 with protocol smtp and e.g. smtp_auth login plain to require authentication (for outbound mail relay by users).
•	IMAP (Port 143) – protocol imap; by default IMAP supports LOGIN and PLAIN. You can also enable CRAM-MD5 if your back-end supports it: e.g. imap_auth login plain cram-md5;[11].
•	POP3 (Port 110) – protocol pop3; e.g. pop3_auth plain apop cram-md5; to allow APOP or CRAM-MD5 in addition to plaintext auth[12].
Each listener can be tuned to match what the internal server supports. Be sure that weak or plaintext mechanisms are only allowed if a secure channel (TLS) is in use.
Defining Upstream Servers: Unlike HTTP proxying where you can use proxy_pass directly in the config, the mail proxy requires an external authentication lookup to determine the correct upstream server for each connection. In fact, configuring an auth service is mandatory – NGINX will consult this service to verify credentials (if any) and get the mail server IP/port to proxy to[13]. This extra step is what lets NGINX route users intelligently. You set the auth service URL via the auth_http directive in the mail context, pointing to a local or external HTTP endpoint[14]. For example:
mail {
    server_name   mail.example.com;
    auth_http     127.0.0.1:9000/auth;   # URL of authentication service
    # ...
}
When a client connects and attempts to log in (or sends the first SMTP command), NGINX will make an HTTP request to this URL with details of the connection. The auth service must respond with appropriate headers indicating Auth-Status (OK or an error) and the chosen Auth-Server and Auth-Port for NGINX to proxy to[15][16]. A successful response looks like:
HTTP/1.0 200 OK
Auth-Status: OK
Auth-Server: 10.0.0.5
Auth-Port: 143
This would instruct NGINX to forward the client to the IMAP server at 10.0.0.5:143[16]. If the auth service returns an error message instead, NGINX will convey that to the client (and optionally how many attempts remain)[17]. Important: Because the mail module’s logic depends on this auth lookup, even if you don’t need to authenticate (say, for unauthenticated SMTP inbound), you still need an auth_http service. In such cases the service can simply return “Auth-Status: OK” and an upstream—possibly using the client’s destination domain to select the mail host. For instance, with smtp_auth none, NGINX will send Auth-Method: none with an empty user/pass to the auth URL[18]. The auth service might parse the Auth-SMTP-Helo or wait for the RCPT TO (passed via headers like Auth-SMTP-To) to decide which internal MX server handles that domain[19][20]. (A beginner’s example is to route *@eng.abc.com to one server and *@hr.abc.com to another based on the recipient domain[21][22].)
External Auth Implementation: Your auth server can be a simple script (in PHP, Python, etc.) or even an existing web service. The NGINX auth protocol (over HTTP) provides headers such as Auth-User, Auth-Pass, Auth-Protocol, Client-IP, and more to the script[15]. The script’s job is to validate the credentials (if a login is attempted) and map the user to an appropriate backend. For example, a PHP auth script might check the username against a database or via an IMAP login attempt to Dovecot, then return the internal server’s IP/port. It could also implement routing logic (e.g. send user Alice to server A, Bob to server B for load distribution or staged migration). If you have only one mail server, the auth service can simply always return that server’s address (after optionally verifying the password). Note that NGINX will not proxy the connection until it gets an “OK” from the auth service[13]. In other words, the actual mail server won’t see the client until NGINX’s auth phase is satisfied.
Tip: For better security, use auth_http_header directives to send a secret token or other custom headers to your auth service[23]. This helps the script verify that requests are actually coming from your NGINX and not an attacker. For example: auth_http_header X-Auth-Key "supersecret"; (Your script would check for this header and value).
NGINX also allows passing certain info to the auth service via custom headers. For instance, you can send X-Auth-Port to indicate which port the client connected to (useful if one auth endpoint handles SMTP vs IMAP logic differently)[24]. In community examples, admins include headers like the listening port or a User-Agent string identifying NGINX’s request[25]. These are optional but can be helpful in complex setups.
TLS/SSL and STARTTLS Configuration
Securing mail traffic in transit is critical. NGINX’s mail module supports both implicit TLS (SSL-wrapped protocols on port 465, 993, 995, etc.) and explicit TLS via STARTTLS (on port 25, 587, 143, 110, etc.)[26]. You can enable SSL/TLS globally for all mail listeners by adding ssl on; inside the mail context[26]. This means any server {} that “speaks” a TLS-enabled protocol (IMAPS, POP3S, SMTPS) or accepts STARTTLS will use SSL. Alternatively, you can control it per listener: within a server {}, use ssl on; for ports that expect an immediate TLS handshake (e.g. listen 993; protocol imap; ssl on;) or use starttls on; for ports that begin unencrypted but may be upgraded[26]. For example, on an SMTP port (25 or 587) you would typically configure starttls on; to allow the STARTTLS command[26]. On port 465 (legacy SMTPS) or 993 (IMAPS), use ssl on; since clients expect SSL from the start.
Certificates: In the mail context, specify the certificate and key to use with ssl_certificate and ssl_certificate_key directives[27]. These should be in PEM format and include any intermediate CA certificates as needed. All mail listeners will use this certificate bundle. If you host multiple mail domains, you have a few options since NGINX’s mail proxy does not support SNI-based virtual hosts (name-based selection of different certs is not available)[28]. You can either use a unified certificate containing all necessary subject alt names (SANs) for your domains, or run separate IPs (or ports) for different certificates[28]. In practice, using a wildcard or multi-domain certificate (e.g. covering mail.domain1.com and mail.domain2.com) is the simplest approach if one IP must serve several domains.
It is a best practice to disable outdated protocols and ciphers on the proxy. Specify ssl_protocols TLSv1.2 TLSv1.3; (dropping older SSLv3/TLSv1.0 which are insecure) and set ssl_ciphers to a strong suite, e.g. HIGH:!aNULL:!MD5 which allows only high-strength ciphers and excludes known weak ones[29]. For example, an NGINX mail config might include:
    ssl_protocols       TLSv1.2 TLSv1.3;
    ssl_ciphers         HIGH:!aNULL:!MD5;
    ssl_prefer_server_ciphers on;
    ssl_session_cache   shared:SSL:10m;
    ssl_session_timeout 10m;
This ensures modern TLS is used and can improve performance by caching TLS session parameters[29][30]. The session cache and timeout settings help clients resume TLS sessions, reducing handshake overhead. Set ssl_prefer_server_ciphers on; to use your cipher ordering (this is often recommended to enforce the strongest cipher available).
For STARTTLS listeners, setting starttls on; makes NGINX advertise the STARTTLS capability in its initial response. It will handle the TLS negotiation with the client when STARTTLS is issued, then continue proxying to the backend over the established secure channel. The certificate and cipher settings above apply in this scenario as well. Make sure your internal servers are configured consistently: e.g. if NGINX handles STARTTLS on SMTP port 25, your back-end Postfix can receive the connection already decrypted (as plain SMTP), or you could even have NGINX initiate a TLS connection to Postfix if you want encryption on the LAN (in which case Postfix would need to expect PROXY protocol or a client cert – typically not necessary on a secure LAN).
Example: The configuration below (from NGINX’s documentation) shows a mail proxy with SSL enabled for all services, and both implicit and explicit TLS on different ports[31][32]:
mail {
    server_name      mail.example.com;
    auth_http        localhost:9000/nginxauth.cgi;
    ssl              on;
    ssl_certificate  /etc/ssl/certs/server.crt;
    ssl_certificate_key /etc/ssl/private/server.key;
    ssl_protocols    TLSv1.2 TLSv1.3;
    ssl_ciphers      HIGH:!aNULL:!MD5;
    proxy_pass_error_message on;

    server {
        listen   25;
        protocol smtp;
        smtp_auth none;
        starttls on;
    }
    server {
        listen   587;
        protocol smtp;
        smtp_auth login plain;
        starttls on;
    }
    server {
        listen   993;
        protocol imap;
        ssl      on;
    }
    server {
        listen   143;
        protocol imap;
        starttls on;
    }
    # ... (POP3 servers similarly)
}
In this example, port 25 is set with smtp_auth none and starttls on (meaning it will allow STARTTLS for inbound SMTP but not require any AUTH)[10], while port 587 allows AUTH mechanisms and also supports STARTTLS. IMAP is offered on 993 (TLS) and 143 (STARTTLS) for client flexibility. All share the same certificate and SSL settings defined at the top.
Ensure you have obtained a valid TLS certificate for your mail endpoint (from Let’s Encrypt or another CA), and keep it updated. Many admins automate certificate renewal and use a script to reload NGINX on renewal. If your mail servers themselves also present certificates (for example, if some clients connect directly internally), you might either configure them with the same cert/key or let NGINX do all TLS and send plaintext internally. The latter is simpler for one controlled environment, as highlighted by users deploying mail servers behind Nginx Proxy Manager (they often copy the LE cert from the proxy to the mail container so both present the same certificate when needed)[33][34].
Reverse Proxy Deployment Between WAN and LAN
Network Topology: In a typical setup, NGINX runs on a DMZ host or gateway with a public IP, and your actual mail servers (Postfix, Dovecot, etc.) run on the internal LAN (with private IPs). The firewall/NAT is configured so that external mail ports point to NGINX. NGINX then connects to the internal servers on their LAN addresses. This isolation means the internal servers are not directly exposed to the internet – all traffic must flow through the proxy. It’s wise to firewall the internal mail servers to only accept connections from NGINX’s host (and perhaps other trusted internal sources). For example, your Postfix might listen on port 25 but blocked at the perimeter except from NGINX; NGINX in turn is the only process talking to Postfix from the outside. This reduces attack surface and ensures any external connection undergoes the proxy’s scrutiny.
Upstream Configuration: You can run the mail proxy on the same machine as the mail server (binding to localhost for backend), but in an edge/LAN scenario they are separate. The Auth-Server and Auth-Port that your auth service returns should be the internal address of the appropriate server. Often, the auth service can be co-located with the mail server (for instance, a script running on the mail server host that knows how to authenticate a user and determine if that user’s mailbox is local). In more advanced setups, the auth service might be a central directory or API (like an LDAP lookup or a small microservice) that has a mapping of usernames to servers.
Multiple Backends: NGINX’s mail proxy is capable of directing different users to different backend hosts – useful for horizontal scaling or multi-tenant hosting. For example, you might host two separate mail systems (e.g. one Postfix/Dovecot stack for companyA.com and another for companyB.com) but want a unified ingress. The auth service could inspect the username or domain (Auth-User or the domain part of it) and respond with the corresponding server IP. NGINX then proxies transparently to the correct internal host. The end-user’s mail client still just uses mail.example.com as the server. This design was even used to migrate mailboxes between servers with no client downtime: one admin describes using NGINX to route users either to an “old” or “new” mail server based on a migration list, allowing a staged cutover without requiring configuration changes by users[35][36].
Note on Multiple Domains: As mentioned, NGINX’s mail module doesn’t do name-based virtual hosts. This means if you service multiple mail domains (e.g. mail.domain1.com and mail.domain2.com) on the same proxy, you generally use one certificate covering both. Alternatively, you could bind multiple IP addresses: for instance, have NGINX listen on IP1 for domain1 (with cert1) and IP2 for domain2 (with cert2). You’d then use the listen [IP]:port and ssl_certificate directives inside separate server blocks bound to those IPs[28]. Using separate ports per domain is another approach but not user-friendly (clients would need non-standard ports), so it’s rarely done. The simplest in practice is a SAN certificate or wildcard that includes all service hostnames.
Internal Service Ports: Often, administrators run the internal mail services on non-standard ports to differentiate proxied traffic. For example, your Dovecot might listen on port 10143 for proxied IMAP, separate from 143 used for any direct connections. This isn’t strictly necessary, but some configurations (like with Dockerized mail stacks) do this to apply different settings (e.g. requiring PROXY protocol on the proxy-facing port, see next section). In your NGINX auth responses, you would then return those internal port numbers. Keep track of these in documentation or config (for clarity, you might define variables for them).
Finally, consider high availability: if you need redundancy, you can deploy multiple NGINX proxy nodes (with a load balancer or DNS round-robin, or VRRP/keepalived for failover). Each would be identically configured to proxy to the backends. The state (being mostly just TCP proxy after auth) is not shared, so HA is mainly about failing over the listener. Make sure the auth service is also highly available or replicated, since it’s critical to new connections.
Mail Authentication and Supported Methods
Properly handling authentication is central to a mail proxy. NGINX itself does not verify user credentials against a mail store; instead, it delegates that to the auth_http service as described. However, you configure what authentication methods (SASL mechanisms) NGINX will accept from clients and how those are passed along.
SASL Mechanisms: Common mechanisms include PLAIN and LOGIN (both transmit a password in base64, relying on TLS for security), CRAM-MD5 (challenge-response hash, avoids sending plain password), and others like APOP for POP3. In NGINX config, the imap_auth, smtp_auth, and pop3_auth directives enumerate which mechanisms to advertise[9]. For instance, imap_auth login plain cram-md5; means an IMAP client can attempt any of those; if the client tries something not listed (e.g. “AUTH GSSAPI”), NGINX will reject it. Configure this list to match what your backend supports and what you want to allow. It’s advisable to permit only secure methods unless absolutely needed. Typically: for SMTP submission, LOGIN and PLAIN are fine (over TLS), and for IMAP/POP, PLAIN (with TLS) or perhaps CRAM-MD5 for legacy clients. If you need OAuth 2.0 token-based auth (XOAUTH2 mechanism, used by some modern clients), note that NGINX Open Source does not natively support XOAUTH2 in the *_auth directives. There are community patches for XOAUTH2[37], but without them NGINX will likely treat an “AUTH XOAUTH2” as an unknown method. In such cases, one workaround is to allow the external method and have the auth script handle it (or bypass NGINX mail proxy for those clients). NGINX Plus as of 2025 does not list native XOAUTH2 support either, so if OAuth2 for IMAP/SMTP is required (e.g. Office365 or Gmail integration), carefully test if the proxy can be in place. Otherwise, clients might need to connect directly or use app-passwords that fall back to basic auth.
Password Verification: Your auth_http service is responsible for checking credentials if the client is logging in. Best practice is to validate the username/password against the same source as the mail server. For example, if Dovecot uses a SQL users table or LDAP, your auth script can query that. If that’s complex, another trick is to let the backend do the auth: have NGINX always return “Auth-Status: OK” (choosing a server) and not truly verify the password in the script. In this mode, NGINX will proxy the connection and the backend mail server will ultimately accept or reject the login. However, you should then enable passing error messages back so that a bad password on the backend is communicated to the client. Use proxy_pass_error_message on; in the mail block to have NGINX relay backend auth failures[38][39]. Without it, if the backend says “invalid password”, NGINX might simply drop the connection or give a generic error. Enabling it ensures the real error (like “Authentication failed”) is shown to the user if the backend rejects the login[39]. Still, the recommended approach is to have the auth service pre-check credentials to avoid spawning an unnecessary backend session on failures. This also allows implementing brute-force protection at the auth layer (for example, you can introduce a delay or failure after X attempts via the Auth-Wait header in responses[17]).
Authentication Flow (simplified): For IMAP/POP3, the client will send a LOGIN command with user and pass. NGINX intercepts that and sends an auth request. If valid, it then establishes a connection to the chosen server. At that point, NGINX typically performs the login to the backend for the user. In fact, in the case of CRAM/APOP, the auth service returns a clear-text password (Auth-Pass) which NGINX uses to authenticate to the upstream[40][41]. For SMTP, the flow can vary: on port 587, clients issue AUTH PLAIN …. NGINX will again call auth_http. If proxy_smtp_auth is off (default), NGINX considers the user authenticated after a successful auth response and will not repeat the AUTH to the backend[42]. Instead, it can use the SMTP extension XCLIENT to inform the backend that the client is authenticated. When xclient on; (the default) is enabled, NGINX will send an XCLIENT command to the SMTP server with details like the real client IP and possibly the logged-in username[43]. Many SMTP servers (Postfix, etc.) treat a connection coming from a trusted proxy with a successful XCLIENT login as an authenticated client, permitting relay[44]. If the backend does not support XCLIENT (for example, Exchange or Exim), another strategy is to enable proxy_smtp_auth on; which causes NGINX to relay the SMTP AUTH command to the backend instead[42]. In that mode, the backend will see and verify the credentials itself (much like a direct connection). However, using proxy_smtp_auth on in combination with TLS and XCLIENT can get tricky – typically you use one method or the other. For most Postfix deployments, the preferred method is to keep proxy_smtp_auth off and use XCLIENT, as it’s more efficient.
Preserving Client Identity: One challenge of any proxy is ensuring the backend sees the actual client’s IP and info (for logging, SPF/DMARC checks, geo-blocking, etc.). NGINX addresses this via two mechanisms: - XCLIENT for SMTP: As noted, NGINX’s mail proxy can pass the client’s IP (and HELO, envelope info) to SMTP servers by issuing an XCLIENT SMTP command during proxy handoff[43]. Postfix and some other MTAs support XCLIENT, but you must configure the MTA to allow it from the proxy’s IP. For Postfix, add NGINX’s IP to the smtpd_authorized_xclient_hosts setting. This tells Postfix to trust XCLIENT from that host and treat the provided IP/host as the real client for logging and access control. With XCLIENT enabled, Postfix will log the true origin IP and apply its usual anti-spam rules (RBL checks, rate limits) based on that data[44][45]. This is vital to prevent the proxy from becoming a blind relay – without it, Postfix would see all traffic as from NGINX’s IP and could inadvertently bypass restrictions. - PROXY Protocol: Another method, applicable to all protocols, is the HAProxy PROXY protocol (v1 or v2). NGINX mail proxy (since 1.19.8) can be configured to send a PROXY header when connecting to upstreams (proxy_protocol on; in the mail/server block)[46]. This is a short line sent before any TLS or protocol handshake, encoding the original client IP and port. The backend must support and be expecting this header. Postfix supports PROXY protocol on inbound connections (e.g., before Postscreen) when appropriately configured (e.g. setting postscreen_upstream_proxy_protocol = haproxy and related options)[47]. Dovecot natively supports proxy protocol on its listeners (you can enable it with the haproxy = yes setting on the imap/pop3 socket). Exim also supports PROXY protocol for inbound SMTP if compiled with SUPPORT_PROXY and configured via hosts_proxy ACL[48][49]. Using PROXY protocol can be simpler than XCLIENT for IMAP/POP3 because those protocols have no equivalent of XCLIENT. For instance, if you proxy IMAP without PROXY, Dovecot will log all logins coming from NGINX’s IP – making fail2ban and location-specific rules ineffective. Enabling proxy protocol ensures Dovecot sees the real remote IP. Community members have reported success using PROXY protocol with NGINX, Postfix, and Dovecot – it’s considered a de facto standard for proxies[50]. To use it, you’d configure NGINX like: listen 143 proxy_protocol; proxy_protocol on; and likewise configure the mail server’s port to expect proxy headers.
Compatibility Notes:
- Postfix: Use XCLIENT if possible (more lightweight). Ensure to restrict NGINX’s IP in mynetworks or authentication policies appropriately – you don’t want Postfix treating NGINX as a fully trusted relay without authentication. Instead, trust it just enough to allow XCLIENT or PROXY info. Also consider enabling Postfix’s smtpd_proxy_filter or postscreen features before acceptance if you need to apply spam filters at the proxy stage – though NGINX itself won’t inspect content, you might integrate a milters or downstream filters.
- Dovecot: Enable proxy protocol on the Dovecot side if using it (e.g. in dovecot.conf: service imap-login { inet_listener imap { port = 143; haproxy = yes; } }). If not using PROXY, you could alternatively use Dovecot’s proxy_ip setting in logging to at least record an X-Forwarded-For header if passed (not common for IMAP). In most cases, PROXY protocol is the robust solution for IMAP/POP.
- Exim: Since Exim doesn’t support XCLIENT[51], PROXY protocol is the way to convey client IP for SMTP. As per Exim’s documentation, you’d list the proxy host in hosts_proxy and ensure Exim is built with proxy support[48][49]. Keep in mind that when using PROXY, the backend sees the real IP only after the proxy header is processed, so initial connection limits should be adjusted. Exim notes that you may need to raise connection limits and instead enforce limits via an ACL on $proxy_external_address to avoid treating the proxy as a single abusive host[52][53].
In summary, to maintain all the usual security and logging of your mail servers, enable XCLIENT and/or PROXY protocol between NGINX and the mail server. This way, things like SPF checks, geo-blocking, fail2ban, and quota tracking continue to work correctly. NGINX’s directives xclient on; (enabled by default) and proxy_protocol on; make this possible[43][46]. (Do not enable proxy_protocol unless your backend is definitely expecting it – otherwise the first line it receives will confuse it.)
Security Hardening Best Practices
Deploying NGINX as an edge mail proxy introduces another layer where security must be enforced. Follow these best practices to harden the setup:
•	Use TLS Everywhere: Ensure all client connections use encryption, either via implicit TLS or STARTTLS. Configure NGINX to require TLS for authentication. For example, on SMTP submission listeners, you can enforce that AUTH is only offered after STARTTLS. NGINX will automatically hide the AUTH capability until TLS is established (if using starttls only; directive – note: starttls on by itself allows optional upgrade; starttls only is an NGINX Plus feature to require TLS). At a minimum, instruct users to use TLS ports (IMAPS, SMTPS) or STARTTLS on standard ports. Use strong certificates and keep them updated.
•	No Open Relay: By fronting SMTP, you must prevent unauthorized relaying. We achieved part of this by disabling SMTP AUTH on port 25 (no client can log in on the MX port)[10]. This means external senders can only use port 25 to deliver mail to local domains (as your Postfix will allow). For port 587, where AUTH is enabled, only authenticated users can send outbound mail. Postfix (or the SMTP backend) should still be configured with relay_domains (or equivalent) so it relays mail only for local users or domains. If using XCLIENT, Postfix will apply its smtpd_recipient_restrictions as usual to deter relaying. Essentially, the presence of NGINX should not change your relay policy: treat all proxied inbound connections as untrusted external and all authenticated submission connections as trusted. Verify that your backend sees the proper authentication status or source IP for these decisions (as covered above). Also, ensure NGINX’s own auth script isn’t allowing an outsider to proxy to arbitrary hosts – it should only direct to valid internal mail servers (so someone can’t use your proxy as a jump host). NGINX’s mail proxy by itself will not queue or forward mail; it just tunnels, but misconfiguration could inadvertently permit a form of open relay if not careful.
•	Brute-Force and Abuse Mitigation: Since NGINX will handle all login attempts, it’s a good choke point to detect and block brute-force password guessing. Unfortunately, NGINX mail module doesn’t have built-in throttling per IP or tarpitting features. However, you can leverage the Auth-Login-Attempt header (which increments on each failed try) and program logic into your auth service to set Auth-Wait (which tells NGINX to delay and limit further attempts)[17]. For example, after 5 bad logins, your auth service could start returning Auth-Wait: 3 with an error, causing NGINX to pause 3 seconds before allowing another attempt. Additionally, implement fail2ban on NGINX’s log or on the mail server logs. If NGINX logs authentication failures (it can log via the error_log when auth_http returns an error or when connections close), those can feed fail2ban to block offending IPs at the firewall. Many prefer to simply monitor the backend (e.g. Dovecot) logs for failed logins since with proxy protocol the real IP is visible. Dovecot and Postfix also have settings to delay responses on failure or shut down connection after X bad attempts – these still apply behind NGINX as long as the backend sees the attempts distinctly (with XCLIENT/PROXY preserving IP). In short, coordinate your security measures: NGINX can handle the encryption and routing, but let the mail servers or auxiliary tools handle content scanning and login abuse detection.
•	Service Hardening: Only enable the ports and protocols you actually need on NGINX. For example, if you don’t offer POP3, don’t configure an NGINX POP3 listener. If you only want secure versions, you might not even listen on 143/110 at all (force users to 993/995). Also consider banner masking – NGINX’s mail module by default sends a generic protocol banner (like * OK IMAP4rev1 or for SMTP it sends the upstream’s banner after connect). It’s possible to customize the IMAP/POP3 capabilities that NGINX advertises pre-login: directives imap_capabilities and pop3_capabilities let you specify exactly what features to advertise[54]. You might want this if your backend has certain extensions (UIDPLUS, IDLE for IMAP, etc.) so that the client is aware of them before login. For example, one configuration sets: imap_capabilities "IMAP4rev1" "UIDPLUS" "IDLE" "LITERAL+" "QUOTA"; to mirror Dovecot’s capabilities[54]. This helps avoid client confusion or unnecessary capability queries. Similarly, smtp_capabilities can adjust the EHLO response if needed (though NGINX usually passes through SMTP features from the backend after connection).
•	Logging and Monitoring: Ensure NGINX’s logging is set appropriately. The mail module logs are not as detailed as HTTP by default, but you can increase the log level for debugging authentication issues. Monitor the NGINX logs for repeated connection attempts or errors contacting the auth service. Also watch your backend mail logs closely after deploying the proxy to confirm that client IPs and auth statuses appear correct (for instance, verify that a failed IMAP login is attributed to the right remote IP, not NGINX’s IP). If not, adjust your proxy protocol or XCLIENT settings.
•	Keep Software Updated: Both NGINX and your mail servers should be kept up-to-date. NGINX mail module bugs are rarer but can be critical (since it deals with raw protocols and authentication, any security bug could be severe). The same goes for your auth service script – if it’s custom code, treat it securely (harden the web server it runs on, use secure coding practices to handle input from NGINX, etc., since it processes credentials in plaintext). Use firewall rules so that only NGINX can talk to the auth service port.
By applying the above measures, you create a robust mail proxy that adds security rather than becoming a new attack vector. The goal is to replicate the security posture of a direct mail server deployment, or improve on it, even though you’ve introduced an additional hop.
Integration Tips for Postfix, Dovecot, and Exim
NGINX’s mail proxy can work in front of all major mail server software. Below are some specific compatibility notes and example configurations for Postfix (MTA), Dovecot (IMAP/POP), and Exim (MTA):
•	Postfix (SMTP) – In a typical setup, Postfix will receive incoming SMTP from NGINX (for MX) and also handle authenticated SMTP for outbound mail. To integrate:
•	Enable the XCLIENT extension on Postfix. In main.cf, set smtpd_authorized_xclient_hosts = <NGINX_IP> (and reload Postfix). This allows NGINX to pass client info. When an SMTP client authenticates at NGINX, NGINX will omit the LOGIN parameter in XCLIENT if it’s handling auth itself[55][43]. Postfix will log the real IP and, if you use Postfix’s smtpd_client_restrictions or RBLs, they’ll act on the client IP as expected.
•	If your NGINX auth_http passes a “authenticated username” through (via XCLIENT’s LOGIN parameter or via setting up the connection as authenticated), ensure Postfix knows to trust the proxy for relaying. A common method is to treat the connection as “sasl authenticated” by using XCLIENT LOGIN or by simply letting Postfix think it’s from a trusted source and not require re-auth. Be careful: do not list the NGINX host in mynetworks, because that would allow any mail via NGINX to relay without auth. Instead, require either XCLIENT with LOGIN or have Postfix still demand SMTP AUTH if no XCLIENT provided. If using proxy_smtp_auth on; (forwarding the AUTH command), then Postfix will directly handle the AUTH as if the client connected, which is also fine – just treat NGINX as a plaintext TCP proxy in that case.
•	Postfix and PROXY protocol: Postfix 3.4+ with Postscreen can accept PROXY protocol. If you prefer that over XCLIENT, set postscreen_upstream_proxy_protocol = haproxy and related options[47], and run Postfix on a separate port (or all ports) that expects PROXY. In practice, many find XCLIENT simpler for SMTP, but PROXY is an option (especially if you already use it for Dovecot).
•	Submission Port Setup: Configure NGINX to listen on 587 and 465 and forward to Postfix’s submission service (usually localhost:587). On those, use smtp_auth login plain in NGINX and have the auth service verify users. On the Postfix side, you might set smtpd_client_authentication = yes and allow relaying for authenticated users (which is standard). If using XCLIENT, Postfix might not even see an AUTH if NGINX consumed it; you can rely on the fact the proxy only forwards mail it authenticated. Alternatively, allow the AUTH to pass through (proxy_smtp_auth on) so Postfix logs the user’s auth attempt directly. Either approach is acceptable, just test that a client can indeed send mail via the proxy.
•	Dovecot (IMAP/POP3) – Dovecot is often the IMAP/POP backend for user mailboxes.
•	Make sure Dovecot’s mail_login_proxy settings are adjusted if needed. If using PROXY protocol, enable it on Dovecot’s listeners (e.g., protocol imap { listen=imap_proxy:143 } in combination with haproxy = yes). Dovecot will then expect a proxy line and parse the client’s IP from it[56][57].
•	If not using PROXY, be aware that all connections from NGINX will show up with NGINX’s IP. Dovecot’s last-login IP tracking and any fail2ban watching Dovecot logs will see the proxy’s IP, which is not ideal. Thus, PROXY protocol is highly recommended in this case (the Docker Mailserver project, for example, uses Traefik/HAProxy with PROXY protocol for exactly this reason, noting that Dovecot and Postfix both support it[58][59]).
•	Authentication: NGINX typically handles the user auth against Dovecot’s user database via the auth_http service. Another approach (if one wanted to simplify) could be to have NGINX always succeed auth and then let Dovecot verify when the proxy attempts login. But as discussed, that can complicate error handling. If you have a large user base, it’s worth tying NGINX’s auth to Dovecot’s auth system – for example, use Dovecot’s checkpassword or REST API (if available) to validate creds. Some setups use a Dovecot proxy as an alternative to NGINX (Dovecot can proxy IMAP to other Dovecot instances), but if NGINX is already in place for multiprotocol, it works fine as described.
•	IMAP/POP3 Capabilities: Ensure NGINX advertises capabilities in line with Dovecot’s. For instance, if Dovecot supports IMAP IDLE (push mail), add "IDLE" to imap_capabilities on NGINX[54]. After user login, NGINX will generally pass through the server’s own capability list (on IMAP, the client typically re-issues CAPABILITY after auth which NGINX will forward to Dovecot to handle). But pre-auth, NGINX’s built-in reply needs to be set. The default NGINX IMAP capabilities might omit some extensions (like IDLE or LITERAL+) until login, so adjust if clients encounter issues.
•	One nuance: Dovecot can perform master-user (proxy) auth or OAuth, but NGINX as a front doesn’t directly support those flows. If you needed OAuth2 for IMAP, you might have to bypass NGINX or implement the OAuth token verification in the auth_http service (which could verify a token against an OAuth server and if valid, return OK to NGINX).
•	Exim (SMTP) – Exim can replace Postfix as the SMTP backend.
•	As mentioned, Exim lacks XCLIENT support[51], so use PROXY protocol to pass client IPs. Compile Exim with SUPPORT_PROXY=yes and configure hosts_proxy = <NGINX_IP> in the Exim config to accept the proxy header[48][49]. When enabled, Exim will expect the proxy line on connections from that host and extract the real IP.
•	Exim’s authentication (PLAIN, LOGIN, etc.) will work similarly to Postfix’s: either let NGINX verify and treat the forwarded connection as already authenticated (perhaps by whitelisting NGINX as allowed to relay without auth – not recommended alone), or forward the AUTH. If you use proxy_smtp_auth on; with Exim, NGINX will send the actual SMTP AUTH command to Exim, and Exim will do its normal user verification (e.g., via Dovecot SASL or an internal user file). This might be simpler if you already have Exim’s auth tied to a database – just let clients authenticate directly through the proxy. In this case, your auth_http service can be very minimal (just always return OK and upstream) since you’re effectively using NGINX purely as a pass-through for SMTP AUTH (though NGINX still needs to know where to route – if only one Exim, that’s easy).
•	Ensure any rate limiting or spam checks in Exim are adjusted. Similar to Postfix, Exim might see a flood of connections from one IP (NGINX). If using PROXY protocol, adjust Exim’s smtp_accept_max_per_host or apply an exemption for the proxy IP, since all connections funnel through it[52]. Then enforce per-IP limits in your ACL using the real IP variable ${sender_host_address} which will contain the client’s IP when proxy protocol is used[60].
Real-World Example: A user had an existing mail server (Postfix/Dovecot) behind NAT and added an NGINX Proxy Manager in front. They forwarded ports 25, 465, 587, 993, etc. to the NGINX proxy, which then sent traffic to the mail server. They encountered an issue where mail clients were seeing a certificate name mismatch – the proxy was presenting a cert for mail.example.com but the internal Dovecot also presented its own certificate to the client, causing confusion[61]. The solution was to have NGINX terminate TLS (so the client only sees NGINX’s cert) and then either use plaintext to Dovecot or install the same cert on Dovecot so it matches[33]. This highlights: decide where TLS is terminated. It’s simplest to terminate at NGINX and use plaintext internally. If you do that, your clients will only ever see NGINX’s certificate (which should be valid for the mail hostname). Make sure the internal services know not to advertise their own TLS in that case (for example, if a client accidentally connected directly to Dovecot’s IP, they might get a self-signed cert – but normally clients won’t). If you choose end-to-end TLS (NGINX passes it through), you lose the ability to do per-user routing or auth at NGINX; NGINX would then be a TCP-level proxy only, which is a different mode (stream proxy) not covered in detail here. Most use cases for a “mail-aware” proxy assume TLS is handled at the edge.
Conclusion and Example Configuration
Bringing it all together, here is a simplified configuration illustrating NGINX as a secure mail proxy for a typical deployment with one Postfix/Dovecot server:
worker_processes auto;

mail {
    server_name mail.example.com;
    auth_http   127.0.0.1:8080/auth;   # internal auth service
    proxy_pass_error_message on;
    ssl on;
    ssl_certificate     /etc/letsencrypt/live/mail.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/mail.example.com/privkey.pem;
    ssl_protocols       TLSv1.2 TLSv1.3;
    ssl_ciphers         HIGH:!aNULL:!MD5;
    ssl_session_cache   shared:SSL:10m;
    ssl_session_timeout 10m;
    # Enable PROXY protocol to backends:
    proxy_protocol      on;
    # (Postfix/Dovecot configured to expect PROXY on their listening ports)

    # SMTP inbound (MX)
    server {
        listen       25   proxy_protocol;
        protocol     smtp;
        smtp_auth    none;
        starttls     on;
        xclient      on;
    }
    # SMTP submission (outbound)
    server {
        listen       587  proxy_protocol;
        protocol     smtp;
        smtp_auth    login plain;
        starttls     on;
        xclient      on;
    }
    # IMAP (TLS)
    server {
        listen       993  proxy_protocol;
        protocol     imap;
        ssl          on;
        imap_auth    login plain;
        imap_capabilities "IMAP4rev1" "UIDPLUS" "IDLE" "LITERAL+";
    }
    # POP3 (TLS)
    server {
        listen       995  proxy_protocol;
        protocol     pop3;
        ssl          on;
        pop3_auth    plain apop;
        pop3_capabilities "TOP" "USER" "UIDL";
    }
}
In this example, NGINX is terminating TLS for all services using a Let’s Encrypt certificate, and proxying to the internal mail server on plain ports (Postfix could listen on 10025 and 100587 with PROXY protocol, Dovecot on 10143/10993, etc., as per your setup). We’ve enabled proxy_protocol on globally in the mail block so that every connection to the backend starts with a PROXY header[46]. The backend Postfix and Dovecot need to be configured to accept that, as discussed. The auth_http service here would handle login checks – for SMTP with no auth (port 25), it will get Auth-Method: none and can immediately return OK with the Postfix’s IP/port. For ports 587/993/995, it will verify user credentials (perhaps by querying Dovecot’s user DB) and respond accordingly. The use of xclient on; ensures that if Postfix sees the connection, it logs real client info[43]. If any error occurs during backend authentication (mailbox full, bad password, etc.), NGINX will pass that error back because we set proxy_pass_error_message on;[39].
This structured approach – dividing responsibilities between the NGINX proxy (TLS termination, routing, basic policy enforcement) and the mail servers (actual mail handling, storage, spam filtering) – can significantly enhance both the security and flexibility of a mail deployment. NGINX’s mail module has been used in scenarios from enterprise Exchange load balancing[62] to small home labs, and its configuration is backed by official documentation and community examples. By following the best practices above, you ensure that your NGINX mail proxy is compatible with major mail software and is configured in a secure, efficient manner[1][36].
