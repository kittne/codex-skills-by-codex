Dockerfile Best Practices
•	Choose minimal, trusted base images: Use lightweight base images from official or verified sources and avoid unnecessary large OS layers[1][2]. Smaller images reduce the attack surface and download faster. Pin specific image versions or digests instead of the latest tag to ensure reproducible builds[3][4]. For example, use FROM python:3.11-slim@sha256:<digest> rather than an unpinned latest.
•	Leverage multi-stage builds for lean production images: Use Docker’s multi-stage build feature to separate build-time dependencies from the final runtime image[5]. In a multi-stage Dockerfile, a first stage can compile or install packages, and a later stage copies only the necessary artifacts into a clean base image. This produces a smaller, safer final image without compilers or dev tools. (Example: Build an app in one stage, then copy the build result into an Alpine runtime image)[6][7].
•	Use .dockerignore to slim the build context: Exclude any files not needed for the image (such as local configs, documentation, or build outputs) by listing them in a .dockerignore file[8]. This prevents sending large or sensitive files to the Docker daemon during build, speeding up builds and avoiding accidental secret leakage.
•	Minimize and combine RUN layers: Install only the packages your app strictly needs and avoid needless utilities (e.g. text editors or network tools)[9][10]. Each additional package can introduce vulnerabilities and bloat. When installing packages in a Debian/Ubuntu-based image, use apt-get install --no-install-recommends and clean up caches in the same RUN command[11]. Combining update+install in one layer and removing temporary files (like /var/lib/apt/lists/*) prevents cache leftovers and keeps the image small[12].
•	Prefer COPY over ADD: The ADD instruction has extra features (like auto-extracting archives and downloading URLs) that can be abused or lead to unintended behavior. Use COPY for simple file transfers into the image to avoid surprises[13]. Reserve ADD only if you explicitly need its advanced functionality.
•	Don’t leak secrets or credentials: Never hard-code secrets (API keys, passwords) in Dockerfiles (for example via ENV or in the image). Anything built into an image layer can be extracted later[14]. Use build-time secrets or ARGs for sensitive data and ensure they aren’t persisted in the final image. At runtime, pass secrets via environment variables or mounted files/volumes, or use Docker/Compose secrets so they aren’t stored in the image[15][16]. For instance, do not ENV AWS_SECRET_KEY=... in a Dockerfile; instead, inject it at runtime or use Compose file secrets.
•	Run as a non-root user: By default, containers run as root, which poses security risks. Include a USER instruction in your Dockerfile to switch to a dedicated user with limited privileges after installing software[17][18]. Create a user/group in the Dockerfile (e.g., RUN addgroup -g 1000 app && adduser -D -u 1000 -G app app for Alpine, or useradd in Debian) and use USER app to drop root rights. This prevents a container escape from granting root access on the host.
•	Add a healthcheck for long-running services: Use the HEALTHCHECK instruction so Docker can monitor the container’s health and restart it if it becomes unhealthy[19]. A healthcheck can run a script or command (for example, an HTTP request to a health endpoint) to ensure the application is responsive. This improves resilience in production by automating recovery from failures. For example:
HEALTHCHECK --interval=30s --timeout=5s --start-period=10s --retries=3 CMD curl -f http://localhost:8080/health || exit 1
•	Keep images updated and rebuild regularly: Treat images as immutable snapshots of a specific time. Update base images and packages frequently to pull in security patches[20]. Use docker build --pull to always get the newest base image updates[21], or enable automated rebuilds. Integrate vulnerability scans into CI so that outdated libraries are caught early[22]. Docker Scout and Trivy can notify you when base images have known vulnerabilities or newer patches[23][24]. Rebuild images often and don’t assume a build from months ago is still secure.
•	Ensure build reproducibility: Pin versions of OS packages and binaries to avoid non-deterministic builds. For example, if using pip, freeze requirements with exact versions. For OS package managers, consider specifying versions or using digest-pinned base images[25]. This makes your Docker builds more repeatable across environments. Additionally, enable Docker Content Trust (DCT) to verify image signatures — by setting DOCKER_CONTENT_TRUST=1, the Docker engine will only pull signed images[26]. This helps ensure you’re using images from trusted sources.
•	Maintain clean layers: Remove any temporary files, build caches, or package manager lists in the same layer they were created. This “layer hygiene” prevents secrets or large files from lingering in intermediate layers. Use multi-stage builds to discard build-only artifacts entirely. You can inspect layers with tools like docker history or the Dive tool to ensure sensitive data isn’t buried in the image. Every layer in an image is permanent once built, so be cautious about what you include.
Docker Compose Best Practices
•	Set resource limits for services: In Compose files, define resource constraints to prevent any single container from hogging the host. For example, under each service use deploy.resources.limits (for v3 Compose in Swarm mode) or mem_limit/cpus (for older Compose) to cap memory and CPU usage[27]. While not all Compose deployments enforce these limits outside Swarm, documenting them is useful for later scaling. At minimum, note memory limits (e.g., mem_limit: 512m) and CPU shares to encourage mindful resource use[27]. This aligns with OWASP guidance to “limit resources (memory, CPU, file descriptors, processes)” for containers[27].
•	Use healthchecks and dependency ordering: Orchestrate multi-container startup by using healthchecks and the depends_on with condition: service_healthy in Compose. This ensures a service waits until its dependency is actually healthy before starting[28][29]. For example, define a healthcheck for the database service, then in the web service use:
depends_on:
  db:
    condition: service_healthy
With this, Compose will wait until the db service passes its healthcheck before launching the dependent service[30]. This prevents race conditions where, say, an app starts before the database is ready.
•	Isolate network access: By default, Compose services on the same network can communicate freely. Create separate networks for components that don’t need to talk to each other, and mark networks as internal (no external access) if appropriate. For instance, you might have a frontend and a backend network – the database joins only the backend network (which is internal), so it isn’t exposed to the internet. Avoid exposing container ports to the host unless necessary; instead, only publish the ports that clients truly need to reach. This reduces the attack surface of your application. OWASP recommends disabling unneeded inter-container communication to harden the environment[31].
•	Manage secrets and sensitive config with Compose secrets: Do not store plaintext secrets in the Compose file or pass them via environment variables if you can avoid it. Use Docker Compose file secrets which mount files at runtime, or use environment variable files that are not committed to source control[16][32]. For example, define a secret at the top-level and reference it in services so that it’s mounted at /run/secrets/<name> inside the container[32][33]. This way, passwords or API keys aren’t visible in docker compose logs or in the image. If Compose secrets aren’t an option (not using Swarm mode), at least use an external config or environment file and add it to .gitignore.
•	Run containers with least privilege in Compose: Ensure that services run as a non-root user and drop unnecessary Linux kernel capabilities. If your Dockerfile didn’t specify a USER, you can use the user: field in the compose service to run it as a lower-privilege UID. Also consider using cap_drop: ["ALL"] and adding back only required capabilities via cap_add for that service (for example, if a container needs to modify network interfaces, add only the NET_ADMIN capability). This follows the principle of least privilege recommended by OWASP (e.g. dropping Linux capabilities and using security profiles)[34]. Additionally, you can set read_only: true for containers that don’t need to write to the filesystem, and use read-only volumes for shared data whenever possible, to mitigate the impact of a compromise[35].
•	Define restart policies: Use restart: in docker-compose to ensure critical services come back if they crash (e.g., restart: "unless-stopped" or always for systemd-like behavior). While not a security practice per se, it improves reliability. In swarm mode (or Docker Compose v2 implementations), you can also specify healthcheck-based restart conditions under the deploy section. This ties into using healthchecks by automatically restarting unhealthy containers.
•	Document and version your Compose file: Treat the Compose file as living documentation of your deployment. Specify image versions/tags for each service (avoid using image: my-service:latest in production) so that deployments are predictable. Use comments to clarify any non-obvious configuration. Keep the file in version control and consider using Compose profiles or multiple files for different environments (dev, prod) rather than one giant file with everything. This keeps your deployment manageable and aligned with best practices (one can use docker compose config to validate and view the final configuration).
Tooling and Automation for Docker Best Practices
•	Lint Dockerfiles with Hadolint: Use Hadolint (a Dockerfile linter) to catch issues and enforce best practices automatically. For example, running hadolint Dockerfile will point out improvements like pinning versions, using --no-install-recommends, and not using root user[36]. Integrating Hadolint into CI or a pre-commit hook helps maintain Dockerfile quality (OWASP also suggests linting Dockerfiles at build time[37]).
•	Build images with BuildKit and proper flags: Use Docker BuildKit for efficient builds and to leverage features like secret mounts. Run docker build with --no-cache and --pull in CI to ensure a clean build with the latest base image[21]. For example:
docker build --pull --no-cache -t myapp:latest .
This updates base layers and avoids using cached outdated packages. You can also use multi-platform builds with docker buildx if needed.
•	Scan images for vulnerabilities: Integrate security scanning into your workflow. Trivy is a great tool to scan container images for known vulnerabilities (CVEs) and misconfigurations. For instance, trivy image myapp:latest will output any CVEs in the image. You can even have Trivy check for compliance with Docker CIS Benchmarks: trivy image --compliance docker-cis-1.6.0 myapp:latest will audit the image against CIS recommendations[38]. Similarly, Docker Scout (the enhanced Docker scanning tool) can be used via the CLI or Docker Desktop UI. Running docker scout cves myapp:latest gives a detailed report of vulnerabilities in the image layers[39], and docker scout quickview provides a high-level health score and recommendations. Use these tools regularly, especially in CI/CD, so that you catch and fix vulnerabilities early[20].
•	Use Docker Compose for development parity: Leverage Compose to mimic production in development. For example, use the same images and environment variables locally and in CI to catch issues early. The docker compose config command can help validate and view the final configuration, and docker compose up --build ensures you’re testing fresh images. Incorporate Compose in your integration tests to spin up the stack and run test suites against it, ensuring all best practices (non-root, minimal images, etc.) work together as intended.
•	Debug and inspect containers effectively: When a container isn’t behaving as expected, use Docker’s tools to troubleshoot. Run an ephemeral debug container with an interactive shell by using docker run --rm -it <image> /bin/sh (or bash) to poke around inside. Use docker logs <container> to check for errors at runtime, and docker exec -it <container> sh to execute commands in a running container for on-the-fly investigation. The docker inspect <container> command will show how the container was configured (ports, env, volumes, etc.), which helps verify that your Compose file or Dockerfile settings took effect. These debugging practices ensure your adherence to best practices hasn’t inadvertently broken functionality, and they help in quickly pinpointing configuration issues.
By following these best practices – drawn from official Docker documentation, the CIS Docker Benchmarks, OWASP security guidelines, and real-world container security wisdom – you can build Docker images and Compose deployments that are efficient, secure, and maintainable. Always automate checks (linting, scanning) and keep up with Docker’s documentation for any new features or deprecations in the container ecosystem[20][40].
