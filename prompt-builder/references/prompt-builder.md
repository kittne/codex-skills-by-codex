Prompt Builder
This skill helps construct and validate robust prompts for large language models in any domain or language. It supports both generating standalone prompt files (e.g. .prompt.md) and producing prompt text inline in chat. The skill operates in two modes – a Builder that crafts the prompt and a Tester that evaluates and refines it – using best practices from leading prompt engineering guides and research. The goal is a clear, effective prompt that reliably yields the desired output across models (a well-designed prompt generalizes to any modern LLM[1]).
Modes and Workflow
Builder Mode: In this phase, the agent gathers requirements and constructs the prompt. It clearly defines the prompt’s purpose, context, instructions, and expected output format. The prompt can be delivered as a saved file or directly in the conversation per user needs. The Builder ensures the prompt is language-agnostic – suitable for natural language or coding tasks without assuming any specific programming language or platform.
Tester Mode: In this phase, the agent verifies the prompt’s quality against success criteria. It may simulate the prompt with sample inputs or analyze it step-by-step to see if it would produce the desired outcome. The agent checks if the prompt instructions are unambiguous and comprehensive. If issues are found (e.g. missing details, incorrect format), the skill switches back to Builder mode to refine the prompt. This Builder→Tester cycle can repeat up to 3 iterations to incrementally improve the prompt. Each iteration uses a research-driven approach: define success criteria and test cases, measure the prompt’s performance, then refine accordingly[2]. If the prompt meets all criteria or after three rounds, the final prompt is output.
Inline vs File Output: If the user requests a prompt file or is working in a repository, the Builder will format the output as a .prompt.md file with appropriate frontmatter or structure. If the user prefers direct answer format, the prompt text is provided in the chat. In either case, the content will be identical – only the delivery changes.
Prompt Design Checklist
When building prompts, the skill follows a structured checklist to ensure all important elements are covered:
•	Purpose: Clearly define the goal and scope of the prompt. State exactly what task the model should accomplish and what problem it solves. A well-defined task prevents ambiguity – ensure the prompt answers “What should the AI do, and what is out of scope?”[2]. For example: “Summarize the following article focusing on key insights and ignoring minor details.”
•	Persona: Establish who the AI should “be” in this context. If helpful, assign the model a role or tone (e.g. “You are a friendly travel advisor,” or “Act as a senior database engineer”). Defining a persona can imbue domain-specific style or expertise. It also sets the appropriate tone, formality, or level of detail for responses. Use the persona to align the model’s voice with user expectations (professional, casual, technical, etc.).
•	Tasks: List the specific tasks or steps the AI should perform. Break down complex requests into clear sub-tasks or instructions. For instance, a prompt might require the AI to 1) gather certain information, 2) analyze or transform it, and 3) produce an answer. Make each required task explicit (bullet lists can help). This reduces confusion and ensures the model addresses every part of the problem. If there is an order or workflow, indicate the sequence (e.g. “First do X, then Y”). The prompt acts like a mini recipe the model should follow.
•	Context: Provide all necessary context and input data. Include any relevant details the model will need (e.g. background info, excerpts of text, data snippets, or user specifics). Use placeholders or variables for dynamic content (e.g. {{user_question}}) when using prompt templates. Ensure the context is sufficient but not cluttered with irrelevant info. If the context is long (like a user-provided text to process), clearly separate it from instructions (for example, precede it with a label like Text: and enclose it in triple quotes or a delimiter[3]). This separation helps the model distinguish instructions from data, preventing confusion or prompt injection[4].
•	Tool Usage: Specify any tools or functions the AI should use, if applicable. If the agent has access to tools (search, calculators, code execution, etc.), instruct when and how to use them. For example: “If the user’s query needs latest information, use the $browser.search tool and cite sources”. Describe tool triggers in the prompt instructions so the model knows the option exists. Conversely, if tool use is not desired, the prompt should clarify that the AI should solve the task without external tools. The instructions should align with the capabilities of the environment (for instance, if using an agent framework like Context7, ensure the prompt respects its tool invocation format).
•	Workflow: Outline the approach or reasoning strategy if needed. For complex tasks, guide the model through a workflow or chain-of-thought. You can instruct the AI to reason step-by-step (e.g. “Consider the problem in stages: first analyze X, then evaluate Y, then conclude Z”). Research shows that explicitly prompting the model to think in steps can improve reasoning quality[5]. If intermediate reasoning or scratchpad outputs are allowed (like in certain agent frameworks), indicate how to use them. Otherwise, the workflow might be an internal plan the model should follow before producing the final answer. Ensure the prompt’s structure encourages the model to stay organized (you can even ask it to output its steps in a structured way if that helps).
•	Outputs: Describe the desired output format and content in detail. Be explicit about what the model’s answer should look like: the format (plain text, Markdown, JSON, code, etc.), length, style, and any sections or headings. If a specific format is needed, show an example in the prompt[6][7]. For instance, “Output the analysis as a JSON object with keys conclusion and evidence” or “Provide the answer in a markdown table.” Include templates or dummy examples: e.g. “Desired format:\n{\n \"conclusion\": \"...\",\n \"evidence\": \"...\"\n}.” When the model sees a template or example output, it’s more likely to follow it[8][7]. Also specify style preferences (formal vs informal, first-person vs third-person, etc.) and any length limits (“in 3-5 sentences”[9]). The more concrete and descriptive the desired outcome, the better the model can satisfy it.
•	Quality Criteria: Define what counts as a successful answer. Establish criteria so the model (and you) can judge the output quality. This might include factual accuracy, completeness (addresses all tasks/questions), coherence, correctness of reasoning, adherence to format, and compliance with any policies (no disallowed content, etc.). For example: “The summary must cover the main points and be under 100 words, with no direct quotes.” Having clear success criteria aids the Tester mode in evaluation[2]. The skill will use these criteria to decide if the prompt’s output is acceptable. If possible, enumerate them as a checklist (e.g. “Output should include at least three key insights and no more than one paragraph of text.”).
•	Examples: Incorporate examples to guide the model (if needed). Few-shot prompting (providing example inputs and outputs) can significantly improve performance on structured tasks[10][11]. If the initial zero-shot prompt isn’t yielding the desired output, the Builder may add one or two examples of the task being done correctly. For instance, for a translation prompt, you might include: “Example: Input: Bonjour → Output: Hello.” Examples help the model infer the pattern or style you want. They can also demonstrate edge cases or tricky formats. Use them judiciously to avoid excessive prompt length – each example consumes tokens, so ensure they’re truly helpful. Even without full few-shot examples, you can still demo the output format as mentioned earlier, or give a partial example (like a step-by-step demonstration of the approach). Always update or remove examples that are not relevant to the current task.
By following this checklist, the prompt will cover why it’s being written, who the AI is in context, what it must do (and not do), what information it has, how it should approach the task, and what output is expected, along with benchmarks for success. This comprehensive approach aligns with proven prompt engineering practices[12].
Best Practices for Effective Prompts
Beyond the checklist, the skill applies general prompt design best practices gathered from expert sources:
•	Place instructions at the top: Always put high-level instructions or role definitions at the beginning of the prompt. If there’s any user-provided text or lengthy context following, separate your instructions from that content with a clear delimiter (such as a line of ### or triple quotes)[3]. This ensures the model first “sees” the directions before any raw data. For example, start with: “You are an AI writing assistant...” followed by “###” or a similar separator, then the context or user query.
•	Be specific and explicit: Vague prompts yield vague responses. Specify every important detail about the task: the context to use, the angle or focus, the output length, format, and style[13]. Instead of “Write a report about climate change,” a better prompt would be: “Write a 3-paragraph report about the economic impacts of climate change in developing countries, in an academic tone, citing two sources.” The latter clearly spells out what’s needed, reducing guesswork. There is no such thing as “too detailed” if the detail is relevant to the task[14]. However, avoid irrelevant or redundant verbosity – every sentence in the prompt should contribute to guiding the model. Strike a balance between clarity and brevity.
•	Use positive instruction instead of prohibitions: Rather than only telling the model what not to do, guide it toward what to do. Prompts framed with negations can backfire because the model may fixate on the forbidden concepts. For example, telling a chatbot “Do not ask the user for personal information” might inadvertently prime it to mention personal information[15]. A better approach is to positively state the rule: “Keep the conversation focused on general assistance and avoid requesting any personal data from the user.” By phrasing it as an affirmative directive (what to do instead of what not to do), you steer the model more reliably[16][17]. In practice, replace rules like “don’t use a casual tone” with “respond in a formal, professional tone,” etc.
•	Favor clarity over cleverness: Simple, clear language in prompts tends to work better than overly complex or figurative wording[18][19]. The goal is to communicate the instructions unambiguously to the model. Avoid metaphorical or ironic instructions that the model might misinterpret. If you need a certain style (like humor or creativity), explicitly request it rather than hoping the model “infers” it. Remember that prompt engineering is often more like giving instructions to a very literal-minded person – be direct and literal about what you want.
•	Optimize for token-efficiency: Prompts consume context window space, so make each token count. Use concise wording once you’ve achieved clarity. Avoid filler phrases and apologies. For instance, “Please, if you don’t mind, could you maybe...” can be simplified to “Generate...” without changing the meaning. That said, don’t omit necessary context or politeness markers if they influence tone – just cut the fat that doesn’t affect outcome. Also prefer reusable variables or references over repeating long texts. If the user’s input or a piece of context is very long, consider summarizing it (if summarization is reliable) or instruct the model to focus on key parts, instead of pasting huge texts in full. A shorter prompt that’s clear will often outperform a verbose prompt full of superfluous content. Iteratively refine wording to trim excess while keeping directives specific[20][21].
•	Provide examples for complex formatting: As noted, showing the model what the output should look like can greatly improve compliance[8]. If you need a list, show a sample list item. If you need JSON, show a skeleton JSON. The model is very good at pattern matching. Just be careful that your example doesn’t accidentally get treated as part of the task input. You can label it clearly as an example or wrap it in delimiters or comments so the model knows it’s illustrative. For instance: “Example – Input: 2+2; Output: 4.” By setting it off, you reduce confusion. The skill tries to include at least one example or format demonstration for prompts that require a very specific output structure.
•	Test incrementally (zero-shot → few-shot): If a straightforward prompt (zero-shot) doesn’t yield satisfactory results, the solution isn’t to immediately write a 500-word prompt. First add one or two example Q&A pairs or demonstrations (few-shot) to see if that helps[10][11]. Often, a couple of well-chosen examples resolve ambiguities. If that still fails, reconsider the prompt wording and the success criteria – are you asking for something the model fundamentally struggles with? In extreme cases where iterative prompt tweaks aren’t enough, it might indicate the need for fine-tuning a model or changing approach[22][23]. Within this skill’s scope, we focus on prompt edits and examples, but it’s useful to know when a prompt might be hitting model limits.
•	Maintain instruction boundaries: Ensure that different parts of the prompt (instruction vs user content vs examples) are clearly delineated, as mentioned earlier. This avoids the model intermixing them. Also, be mindful of system/developer messages versus user messages if working with role-based prompting (like OpenAI’s chat format). Put developer instructions (system-level guidance) in the system role, and user query or data in the user role or a clearly marked section[24]. This helps the model prioritize developer instructions properly[25]. The skill-generated prompts will typically present the structured pieces in an organized way, using headings or separators so that any consuming system (OpenAI, Anthropic, etc.) can easily apply them to the right message fields.
•	Cross-check for compliance and safety: Before finalizing, the skill checks the prompt against AI policy and safety considerations. If the prompt asks the model to do something against content guidelines (even inadvertently), that’s a problem – e.g., a prompt that says “Explain how to do X illegal thing” should be reworked or avoided. Make sure the prompt includes instructions to refuse or safe-complete if the user input might lead to disallowed content. Also consider bias or ethical issues: if the prompt sets a persona or scenario, does it inadvertently prompt the model to produce stereotypes or sensitive content? Adding a line like “The assistant should follow OpenAI content guidelines and not produce disallowed content.” can be wise for safety, though the model’s built-in filters often handle this. In essence, treat the prompt as you would any piece of code or content – review it for unintended consequences.
•	Confirm destructive actions: If the prompt involves agent actions that could be destructive (like deleting files, modifying databases, or any irreversible operation), incorporate a confirmation step. This is critical in agent workflows where the AI can execute commands. For example, instruct the model: “Before performing any destructive action (e.g. deleting data), explicitly ask the user to confirm with a yes/no prompt and wait for approval”. This pattern has been highlighted as essential when connecting AI agents to real systems[26]. The prompt should enumerate what counts as “destructive” in context (deletions, irreversible changes) and model the confirmation dialog. Never assume the AI will automatically know to double-check. Always bake that safeguard into the prompt instructions for safety.
By adhering to these best practices, the skill aims to produce prompts that are clear, specific, and fail-safe. They leverage known techniques (like instruct-first format[3], example-led formatting[27], step-by-step reasoning cues[5], and positive phrasing[16]) that have been shown to improve model responses. The resulting prompt will serve as a strong blueprint for the model’s behavior.
Testing and Iteration
In Tester mode, the skill validates the prompt using the defined quality criteria. This may involve one or more of the following strategies:
•	Dry-run simulation: The agent can perform a mental walkthrough: “If the model sees this prompt and a typical input, what will it do?” It checks if any instruction might be misunderstood. For example, if the prompt says “list three items” but the context has five relevant items, will the model arbitrarily drop two? Such issues are flagged for revision.
•	Sample test cases: The agent might generate or use example inputs to test the prompt. For instance, if the prompt is meant to summarize articles, the Tester can take a short sample article and see if the prompt (when followed) would yield a correct summary. The skill may not always execute an API call, but it can use its own reasoning to predict the outcome or even ask itself to produce a hypothetical output given the prompt. It then checks that against the success criteria (did the output follow format? contain all required info? etc.). This approach of creating evaluation test cases is akin to unit-testing the prompt[2].
•	Criteria checklist review: The Tester goes through each quality criterion and ensures the prompt addresses it. If a criterion is “must cite sources for factual claims,” the Tester verifies that the prompt actually instructed the model to cite sources. If something is missing (like no instruction about citations), that’s a prompt bug to fix in the next Builder iteration.
•	Edge case analysis: The Tester considers edge scenarios: “What if the user input is empty? What if it’s extremely long? What if it contains special characters or potentially problematic content?” If the prompt doesn’t handle an edge case (e.g., not instructing the model what to do if input is missing or if content is sensitive), the skill notes this and will address it in refinement.
After testing, the skill enters the refinement loop if needed. In refinement, it modifies the prompt based on what the tests revealed – perhaps adding an example, rephrasing an unclear instruction, tightening a format specification, or adding a safety disclaimer. It then tests again. This loop continues for up to three cycles or until the prompt meets all success criteria satisfactorily.
Throughout this process, the skill keeps track of changes to avoid going in circles. Each iteration should meaningfully improve the prompt. If by the third iteration the prompt is still failing key criteria, the skill will output the best attempt along with notes (if appropriate) on remaining issues, acknowledging that further tuning might be needed outside of this automated loop.
Examples of Good vs. Bad Prompts
To illustrate the principles, here are brief examples highlighting improvements from a poor prompt to a better prompt:
•	Example 1 – Summarization Prompt
Bad: “Summarize the text below.”
Why it’s bad: Too short and unspecific. Summarize in how much detail? Focus on what aspects? What format? The model might produce a generic summary or even just restate the text.
Better: “Summarize the following article in 2-3 sentences, focusing on the main findings and conclusion. Omit minor details and background information. Output the summary as a single paragraph in a neutral, third-person tone.”[13]
Why it’s better: It specifies length (2-3 sentences), scope (main findings and conclusion only), what to exclude (minor details), the desired tone (neutral, third-person), and format (single paragraph). The model now has clear guidance on how to summarize.
•	Example 2 – Chatbot Prompt
Bad: “You are a customer service bot. Don’t be rude or ask for personal info.”
Why it’s bad: It sets a role, which is good, but the instructions are negatively phrased (“don’t do X”) and incomplete about what to do. Also, “rude” is vague – what is considered rude? No guidance on actual task or style beyond not doing something.
Better: “You are a customer service assistant for an e-commerce company. Always respond in a polite, helpful manner. Do not request personal data like passwords or credit card numbers. If the user’s query is about an order status, ask for the order number. If it’s a return/refund issue, explain the return policy clearly. For other questions, provide a concise and accurate answer or escalate to a human agent if unsure. Always apologize if the company caused an inconvenience, and maintain a friendly tone.”[16][17]
Why it’s better: It positively instructs (“respond in a polite, helpful manner”) instead of just saying “don’t be rude”. It explicitly forbids personal data requests (still needed, but now as one rule among many). It covers specific common scenarios (order status, refund) with directions, making the bot’s behavior more deterministic. It also adds a general rule for unknown cases. The tone and style expectations are clearly stated. This prompt gives the chatbot a much more complete policy to follow, likely leading to consistent and safe outputs.
•	Example 3 – SQL Query Prompt
Bad: “Generate an SQL query for the following: we have tables employees and departments and we need a report.”
Why it’s bad: Very vague – what report exactly? Which fields? Any conditions? The model might guess incorrectly. Also no example or schema given (assuming the model somehow knows or guesses).
Better: “You are a SQL assistant with access to a database. Write a SQL query to find the names of employees and the name of their department, for all employees in the Sales or Marketing departments. Employees table columns: id, name, department_id; Departments table columns: id, department_name. Output just the employee name and department name. Only include employees where department_name is 'Sales' or 'Marketing'. Order the results by employee name.”
Why it’s better: It explicitly states the desired query and result: join employees with departments, filter by department name in a set, select specific columns, order by name. It even provides the schema details, which the model might not fully need but it removes ambiguity about column names. The output expectation (which columns to show) is stated. This prompt leaves little doubt as to what the SQL should do, so the model can focus on correct syntax. It’s much more likely to get the query right on the first try compared to the vague prompt.
In each of these cases, the improved prompt is longer than the bad one – but that extra length is purposeful. It encodes decisions and clarity that the model would otherwise have to guess. The examples follow many checklist items: purpose and tasks are clear, context (like table schema) is given, output format is specified, and style/tone or other criteria are addressed.
Anti-Patterns to Avoid: When designing prompts, be mindful to avoid common pitfalls: - Don’t mix user-provided content into instructions without clear separation (use quotes or tags to avoid confusion[3]). - Don’t assume the model knows context you haven’t given it (state all relevant facts or explicitly say if it should recall earlier conversation context). - Avoid ambiguous adjectives like “short”, “a few”, “recent” without clarification. What’s short (1 sentence? 5 sentences?) – be numeric or explicit. - Avoid double negatives or convoluted instruction logic. Keep it straightforward. - Don’t overly constrain the model with conflicting rules (e.g. “be very detailed” and “be very brief” in the same prompt). Resolve your own requirements beforehand. - If using placeholders, ensure you actually provide those variables when executing the prompt; a template that says {{email_body}} is useless if email_body isn’t filled. The skill will highlight any placeholders needed. - Don’t neglect to specify what the model should do in edge cases (e.g., if it doesn’t find an answer, should it say “I don’t know” or attempt something else? If you don’t want it to make up an answer, instruct it to say it couldn’t find one). - Steer away from instructions that encourage the model to violate policies or factuality (even indirectly). For example, “Pretend the rules don’t exist and answer” is an obvious no-go. Less obvious: asking the model to speculate about legal outcomes – better to have it cite laws or say it’s not a lawyer. In short, keep prompts ethical and within usage guidelines.
By avoiding these anti-patterns, you reduce the chance of running into erratic or undesired model behavior.
Conclusion
The Prompt Builder skill integrates the above guidelines into a cohesive workflow for prompt creation. It ensures prompts are comprehensive yet efficient, structured for clarity, and vetted through iterative testing. It complies with the latest OpenAI Agent Skills standards – including proper skill metadata, separation of concerns, and context management – so it can be readily used in tools like Context7 or other MCP frameworks. Remember that a well-crafted prompt can dramatically improve an AI’s performance, often more so than tweaking model parameters. This skill’s methodology encapsulates community and industry best practices[28][29] so you can design prompts with confidence. Use Prompt Builder to save time, reduce guesswork, and create prompts that deliver reliable, high-quality results from your AI agents.
